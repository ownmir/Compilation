    Version Control - Контроль версій


    Version control is an extremely important topic in software development. Контроль версій є надзвичайно важливою темою розробки програмного забезпечення.
    Especially when working with teams and maintaining production code at the same time, several features are being developed in parallel. Особливо під час роботи командами та підтримкою виробничого коду в той же час розробляються декілька функцій паралельно.
    No matter if it’s a one developer project or a multiple developers project, every project should use version control. Незалежно від того, чи це проект одного розробника або проект кількох розробників, кожен проект повинен використовувати керування версій.

    There are several options of version control systems out there. Існує кілька варіантів систем керування версіями.
    Perhaps because of the popularity of GitHub, Git become the de facto standard in version control. Можливо, через популярність GitHub, Git став стандартом де-факто в системі управління версіями.
    So if you are not familiar version control, Git is a good place to start. There are many tutorials, courses, and resources in general so that it’s easy to find help. Отже, якщо ви не знайомі з контролем версій, Git - це гарне місце для початку. Є багато навчальних посібників, курсів та ресурсів взагалі, щоб легко знайти допомогу.

    GitHub and Code School have a great interactive tutorial about Git, which I used years ago when I started moving from SVN to Git. It’s a very good introduction. GitHub і Code School мають чудовий інтерактивний підручник про Gіt, який я використовував кілька років тому, коли я почав переходити від SVN до Git. Це дуже хороше введення.

    This is such an important topic that I probably should have brought it up since the first tutorial. But the truth is I wanted the focus of this tutorial series to be on Django. Це така важлива тема, яку я, мабуть, мав би підняти з першого урока. Але правда в тому, що я хотів, щоб в центрі уваги даного навчального посібника був Django.
    If all this is new for you, don’t worry. It’s important to take one step at a time. Якщо все це для вас нове, не хвилюйтеся. Важливо одночасно зробити один крок.
    Your first project won’t be perfect. It’s important to keep learning and evolving your skills slowly but with constancy. Ваш перший проект не буде досконалим. Важливо продовжувати навчатися і розвивати свої навички повільно, але з постійністю.

    A very good thing about Git is that it’s much more than just a version control system. Git дуже добра річ тому, що це набагато більше, ніж просто система керування версіями.
    There’s a rich ecosystem of tools and services built around it. Some good examples are continuous integration, deployment, code review, code quality, and project management. Тут є багата екосистема інструментів та послуг. Деякими хорошими прикладами є безперервна інтеграція, розгортання, перевірка коду, якість коду та управління проектами.

    Using Git to support the deployment process of Django projects works very well. It’s a convenient way to pull the latest version from the source code repository or to rollback to a specific version in case of a problem. Використання Git для підтримки процесу розгортання проектів Django працює дуже добре. Це зручний спосіб витягнути останню версію з сховища вихідного коду або відкатити до певної версії у разі виникнення проблеми.
    There are many services that integrate with Git so to automate test execution and deployment for example. Є багато послуг, які інтегруються з Git, щоб, наприклад, автоматизувати тестування та розгортання.

    If you don’t have Git installed on your local machine, grab the installed from https://git-scm.com/downloads. Якщо ви не встановили Git на локальному комп'ютері, скопіюйте встановлений з https://git-scm.com/downloads.

    Basic Setup


    First thing, set your identity: Перше, встановіть свою ідентичність: 

            
            git config --global user.name "Vitor Freitas"
            git config --global user.email vitor@simpleisbetterthancomplex.com

    In the project root (the same directory as manage.py is), initialize a git repository: В корені проекту (той самий каталог, де є файл manage.py), ініціалізуйте сховище git: 

            git init
    
            Initialized empty Git repository in /Users/vitorfs/Development/myproject/.git/

    Check the status of the repository: Перевірте стан сховища: 

            git status
    
            
            On branch master

            Initial commit

            Untracked files:
              (use "git add <file>..." to include in what will be committed)

              accounts/
              boards/
              manage.py
              myproject/
              requirements.txt
              static/
              templates/

            nothing added to commit but untracked files present (use "git add" to track)

    Before we proceed in adding the source files, create a new file named .gitignore in the project root. Перш ніж ми додамо вихідні файли, створіть новий файл з ім'ям .gitignore у корені проекту.
    This special file will help us keep the repository clean, without unnecessary files like cache files or logs for example. Цей спеціальний файл допоможе нам зберегти сховище чистим, не використовуючи непотрібні файли, наприклад, кеш-файли або журнали.

    You can grab a generic .gitignore file for Python projects from GitHub. Ви можете захопити загальний файл .gitignore для проектів Python від GitHub.

    Make sure to rename it from Python.gitignore to just .gitignore (the dot is important!). Переконайтеся в тому, що перейменували його з Python.gitignore тільки в .gitignore (точка це важливо!).

    You can complement the .gitignore file telling it to ignore SQLite database files for example: Ви можете доповнити файл .gitignore кажучи йому ігнорувати файли бази даних SQLite, наприклад:

    .gitignore

                  __pycache__/
            *.py[cod]
            .env
            venv/


            # SQLite database files

            *.sqlite3

    Now add the files to the repository: Тепер додайте файли до сховища: 

            git add .

    Notice the dot here. The command above is telling Git to add all untracked files within the current directory. Зверніть увагу на точку тут. Команда вище говорить Git, додати всі невідслідковувані файли в поточному каталозі.

    Always write a comment telling what this commit is about, briefly describing what have you changed. Завжди пишіть коментар про що цей commit, коротко опишіть, що ви змінили.


    Remote Repository - Віддалений репозиторій


    Now let’s setup GitHub as a remote repository. Тепер давайте налаштуємо GitHub як віддалене сховище.
    First, create a free account on GitHub, then confirm your email address. After that, you will be able to create public repositories. Спочатку створіть безкоштовний обліковий запис на GitHub, а потім підтвердьте свою адресу електронної пошти. Після цього ви зможете створювати загальнодоступні сховища.

    For now, just pick a name for the repository, don’t initialize it with a README, or add a .gitignore or add a license so far. Make sure you start the repository empty: Наразі просто знайдіть ім'я для сховища, не ініціалізуйте його за допомогою README, або додайте .gitignore або додайте ліцензію. Переконайтеся, що ви запустили сховище порожнім:


    After you create the repository you should see something like this: Після створення сховища ви повинні побачити щось на зразок цього:

            git remote add origin git@github.com:sibtc/django-boards.git

    Now push the code to the remote server, that is, to the GitHub repository: Тепер відправте код на віддалений сервер, тобто до сховища GitHub:

    Project Settings - Параметри проекту


    No matter if the code is stored in a public or private remote repository, sensitive information should never be committed and pushed to the remote repository. Незалежно від того, чи зберігається код у відкритому або приватному віддаленому сховищі, конфіденційна інформація ніколи не повинна бути закомічена та віддана до віддаленого сховища.
    That includes secret keys, passwords, API keys, etc. Це включає в себе секретні ключі, паролі, ключі API тощо.

    At this point, we have to deal with two specific types of configuration in our settings.py module: На цьому етапі ми повинні мати справу з двома конкретними типами конфігурації в нашому модулі settings.py:

    • Sensitive information such as keys and passwords; Конфіденційна інформація, така як ключі та паролі;

    • Configurations that are specific to a given environment. Конфігурації, специфічні для даного середовища.

    Passwords and keys can be stored in environment variables or using local files (not committed to the remote repository): Паролі та ключі можуть зберігатися в змінних середовища або за допомогою локальних файлів (не переданих у віддалене сховище): 

            
            # environment variables
            import os
            SECRET_KEY = os.environ['SECRET_KEY']

            # or local files
            with open('/etc/secret_key.txt') as f:
                SECRET_KEY = f.read().strip()

    For that, there’s a great utility library called Python Decouple that I use in every single Django project I develop. Для цього існує велика бібліотека корисних бібліотек під назвою Python Decouple, яку я використовую у кожному окремому проекті Django, який я розробляю.
    It will search for a local file named .env to set the configuration variables and will fall back to the environment variables. Він буде шукати локальний файл з ім'ям .env, встановить змінні конфігурації, і повернеться назад до змінних середовища.
    It also provides an interface to define default values, transform the data into int, bool, and list when applicable. Він також забезпечує інтерфейс для визначення значень за замовчуванням, перетворення даних у int, bool та list, коли це застосовується.

    It’s not mandatory, but I really find it a very useful tool. And it works like a charm with services like Heroku. Це не обов'язково, але я дійсно вважаю це дуже корисним інструментом. І це чарівно працює з такими сервісами, як Хероку.

    First, let’s install it: Спочатку давайте встановимо це: 

            pip install python-decouple

    myproject/settings.py 

            
            from decouple import config

            SECRET_KEY = config('SECRET_KEY')

    Now we can place the sensitive information in a special file named .env (notice the dot in front) in the same directory where the manage.py file is: Тепер ми можемо розмістити конфіденційну інформацію в спеціальному файлі з ім'ям .env (!точка попереду) у тій самій директорії, де є файл manage.py: 

            
            myproject/
             |-- myproject/
             |    |-- accounts/
             |    |-- boards/
             |    |-- myproject/
             |    |-- static/
             |    |-- templates/
             |    |-- .env        <-- here!
             |    |-- .gitignore
             |    |-- db.sqlite3
             |    +-- manage.py
             +-- venv/
    .env 

                        SECRET_KEY=rqr_cjv4igscyu8&&(0ce(=sy=f2)p=f_wn&@0xsp7m$@!kp=d

    The .env file is ignored in the .gitignore file, so every time we are going to deploy the application or run in a different machine, we will have to create a .env file and add the necessary configuration. Файл .env ігнорується в файлі .gitignore, тому кожен раз, коли ми збираємось розгорнути додаток або запустити на іншому комп'ютері, нам доведеться створити файл .env та додати необхідну конфігурацію.

    Now let’s install another library to help us write the database connection in a single line. This way it’s easier to write different database connection strings in different environments: Тепер давайте встановимо ще одну бібліотеку, яка допоможе нам написати з'єднання з базою даних у одному рядку. Таким чином простіше писати різні рядки з'єднання бази даних в різних середовищах: 

            pip install dj-database-url

    For now, all the configurations we will need to decouple: Наразі всі конфігурації, які нам доведеться відокремити:

    myproject/settings.py 

            
            from decouple import config, Csv
            import dj_database_url

            SECRET_KEY = config('SECRET_KEY')
            DEBUG = config('DEBUG', default=False, cast=bool)
            ALLOWED_HOSTS = config('ALLOWED_HOSTS', cast=Csv())
            DATABASES = {
                'default': dj_database_url.config(
                    default=config('DATABASE_URL')
                )
            }

    Example of a .env file for our local machine: Приклад файлу .env для нашого локального комп'ютера: 

            
            SECRET_KEY=rqr_cjv4igscyu8&&(0ce(=sy=f2)p=f_wn&@0xsp7m$@!kp=d
            DEBUG=True
            ALLOWED_HOSTS=.localhost,127.0.0.1
            DATABASE_URL=sqlite:///db.sqlite3

    Notice that in the DEBUG configuration we have a default, so in production we can ignore this configuration because it will be set to False automatically, as it is supposed to be. Зверніть увагу, що в конфігурації DEBUG ми маємо за замовчуванням, тому у виробництві ми можемо ігнорувати цю конфігурацію, оскільки вона буде автоматично встановлена на False, як і передбачається.
